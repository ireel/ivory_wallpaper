# ivory_wallpaper

一个用于生成适配 Lively Wallpaper 的网页壁纸项目。

## Rust Runtime（当前阶段）

已新增 Rust 原生运行器，用于加载并渲染本项目的 `index.html`，为后续本地文件系统与命令能力扩展做准备。

### 文件位置

- `Cargo.toml`
- `src/main.rs`

### 本地运行

```bash
cargo run -- --mode lively .\index.html
```

或直接挂到桌面壁纸层（显示原生图标）：

```bash
cargo run -- --mode workerw .\index.html
```

说明：

- 不传参数时，程序会自动尝试在“可执行文件目录”或“当前目录”查找 `index.html`。
- `--mode lively`：普通应用窗口，交给 Lively Application Wallpaper 接管（推荐）。
- `--mode workerw`：直接挂到桌面 `WorkerW/Progman` 层，不依赖 Lively。
- `--mode fullscreen`：调试用全屏覆盖模式。

### 在 Lively 中尝试

1. 先构建可执行文件：
```bash
cargo build --release
```
2. 在 Lively 使用“Application Wallpaper”模式添加：
- `target\release\ivory_wallpaper_runtime.exe`
3. 让 exe 与 `index.html / styles.css / script.js` 保持同目录（或启动参数显式传入 HTML 路径）。
4. 启动参数建议用 `--mode lively`，不要使用 `fullscreen`。

> 这是第一步“先跑起来”。后续我们可以在 Rust 层加入本地文件读写、命令执行和受控 IPC。

## 当前已实现（MVP）

基于你的需求，当前版本已经实现以下能力：

- 原生桌面图标对齐网格：
  - 取消独立的“desktop grid 图标窗口”。
  - 壁纸仅提供全屏单元格参考线，你可以直接摆放系统原生图标。
  - 默认基准参数为 `2560x1600（150% 缩放） -> 77 / 98 / 72 / -7`（宽/高/水平偏移/垂直偏移）。
  - 支持“网格微调”：基准分辨率、单元格宽高、X/Y 偏移和透明度可调。
  - 在其它分辨率下按基准分辨率同比换算，并结合 `devicePixelRatio` 适配系统缩放。
- 背景图片可更换：右上角“更换背景”按钮会弹出选择窗口，可选预设背景或上传本地图片。
- 时间显示区块：位于顶部中间，显示当前时间和日期，自动实时更新。
- 中下方大区域：将 Markdown 备忘录和 TodoList 合并在同一块信息面板中。
- 备忘录区域：支持编辑 Markdown 并切换预览渲染。
- TodoList：支持新增、勾选完成、删除。
- 配置导入导出：支持保存/加载 JSON 配置，包含壁纸、网格参数、备忘录和待办。
- 本地持久化：网格参数、备忘录、待办保存在 `localStorage`，自定义大图背景优先保存在 `IndexedDB`（避免容量限制）。

## 说明与限制

- 网页壁纸无法直接读取“桌面图标当前坐标”，所以不能自动 100% 吸附原生图标位置。
- 当前实现通过系统图标间距参数初始化网格，再提供手动偏移与尺寸微调来贴合你的实际布局。
- 导出配置时会把自定义背景转成 DataURL 写入 JSON，图片越大导出文件也会越大。

## 快速开始

当前项目是纯静态网页，无需安装依赖：

1. 直接打开项目根目录下的 `index.html`。
2. 或使用任意静态服务器在项目目录启动后访问页面。

## Lively Wallpaper 使用方式

1. 打开 Lively Wallpaper。
2. 选择添加网页壁纸（URL / HTML）。
3. 将本项目的 `index.html` 导入为本地网页壁纸。
4. 设置为桌面后，即可在页面上进行背景、图标布局、备忘录和 Todo 的调整。

## 项目目标

这个项目的目标不是单纯“写一个网页”，而是建立一个能够持续演进的壁纸生成器：

- 输入是主题配置、组件配置、资源文件。
- 输出是一个可以直接被 Lively Wallpaper 加载的网页壁纸目录或网页入口文件。
- 后续可以逐步增加时钟、天气、待办、动态背景、音频响应、系统信息等能力。

## 为什么推荐增量式架构

这个项目很适合采用“增量式架构”而不是一开始做成大而全的系统，原因很直接：

- 壁纸项目的核心交付物其实很小，最终通常只是一个网页入口和一组静态资源。
- 视觉表现、动画、数据源、配置系统都会频繁变化，过早设计复杂平台容易拖慢迭代。
- Lively Wallpaper 本质上是承载网页运行时，因此我们优先保证“可运行、可预览、可导出”，再逐步叠加能力最稳。
- 很多功能天然适合分层扩展，例如先做静态壁纸，再做动态组件，再做可配置化，再做外部数据接入。

换句话说，这个项目最合适的方向不是“先做完整框架”，而是“先做最小可运行内核，再按能力逐层增长”。

## 推荐的总体架构

我建议采用下面这套四层结构：

### 1. 壁纸运行层

职责：真正运行在 Lively Wallpaper 中的网页。

- 产物是静态网页资源，例如 `index.html`、`assets/*`、样式和脚本。
- 这一层只关心展示、动画、交互和性能。
- 默认不依赖后端服务，优先保证离线可运行。

这一层应该尽量薄，只负责“把配置渲染成壁纸”。

### 2. 配置与主题层

职责：描述壁纸长什么样，而不是直接写死在页面里。

- 使用 JSON 或 TypeScript 对象定义主题。
- 配置项可以包括背景、字体、布局、组件开关、动画强度、颜色方案等。
- 后续如果要对接 Lively 的属性面板，也可以从这一层扩展出去。

这一层是项目可维护性的关键。越早把“样式内容”和“程序逻辑”分开，后面越容易增加主题和预设。

### 3. 组件与场景层

职责：把壁纸拆成可独立启用的模块。

建议把页面拆成两类：

- 场景模块：例如纯色极简、玻璃拟态、自然动态背景、信息仪表盘。
- 组件模块：例如时钟、日期、天气、待办、系统监控、音乐状态。

这样后续做新壁纸时，不需要重写整个页面，只需要复用场景布局和组件模块。

### 4. 构建与导出层

职责：把开发态内容整理为可交付给 Lively Wallpaper 的产物。

- 负责编译、打包、拷贝静态资源。
- 负责生成最终目录，例如 `dist/`。
- 如果后续需要，也可以在这一层补充 Lively 所需的描述文件和预设导出逻辑。

这一层让“开发用页面”和“最终壁纸产物”分离，方便调试，也方便以后扩展成多套壁纸输出。

## 推荐的技术路线

对于这个项目，我更推荐下面这种偏轻量、可持续扩展的技术组合：

- `Vite`
  作为开发服务器和打包工具，启动快，适合网页壁纸项目。
- `TypeScript`
  用来约束配置结构、组件接口和导出流程，后期比纯 JavaScript 更稳。
- `CSS Variables`
  用来承载主题系统，颜色、阴影、透明度、字体尺寸都可以配置化。
- `Canvas / SVG / 原生 CSS 动画`
  用于动态背景和轻量视觉效果。
- `JSON 配置文件`
  用于定义预设主题和组件开关。

除非后续确认需要复杂的数据流和大量交互，否则不建议一开始就引入过重的状态管理方案。

## 增量式迭代路线

我建议按下面 4 个阶段推进：

### Phase 1: 最小可运行壁纸

目标：先产出一个能在 Lively Wallpaper 中跑起来的网页。

- 一个固定布局页面。
- 一个静态或轻量动态背景。
- 一个时钟或日期组件。
- 可以本地开发预览，并导出到 `dist/`。

这一阶段只验证“网页壁纸可运行”。

### Phase 2: 配置驱动

目标：让壁纸不再靠手改代码，而是靠配置切换主题。

- 提取主题配置。
- 提取组件开关。
- 支持多个预设。
- 把视觉参数转成 CSS Variables 或统一 token。

这一阶段只验证“同一套代码能生成不同风格”。

### Phase 3: 模块化组件

目标：让功能可以自由拼装。

- 把时钟、天气、信息卡片等拆成独立模块。
- 给组件定义统一输入接口。
- 支持按场景组合组件。

这一阶段只验证“不同壁纸能复用同一套部件”。

### Phase 4: 导出与平台适配

目标：让项目真正成为“壁纸生成器”。

- 自动导出可发布目录。
- 增加预览图、元信息、主题包。
- 视需要补充 Lively 相关配置文件。

这一阶段才开始关注“交付体验”和“多主题分发”。

## 架构原则

为了避免这个项目后期失控，我建议在 README 中明确下面几个原则：

- 先静态，后动态：先保证可运行，再增加实时数据。
- 先配置，后抽象：先把变化点提取出来，再决定是否沉淀成框架。
- 先单主题跑通，后多主题复用：先验证最小闭环，再做主题系统。
- 先离线能力，后联网能力：桌面壁纸优先保证稳定和低耦合。
- 先轻运行时，后复杂功能：壁纸长时间驻留桌面，性能和资源占用很重要。

## 一句话结论

如果这个项目的目标是“生成适配 Lively Wallpaper 的网页壁纸”，那么最合适的不是一开始做复杂平台，而是采用：

**静态网页运行层 + 配置层 + 组件层 + 导出层** 的增量式架构。

这套方式最适合从一个可运行的壁纸快速起步，并在不推翻已有代码的前提下逐步演进成真正的壁纸生成器。
